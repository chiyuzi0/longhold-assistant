# Agent Loop — 受控状态机设计

## 1. 核心原则

LongHold 的 Agent Loop **不是 ReAct 模式**。不搞"思考→行动→观察→再思考"的无限循环。每个任务必须在一组硬约束下完成。

约束：
- 最大步数限制 (default: 12)
- 超时限制 (default: 300s)
- LLM 调用次数限制 (default: 6)
- 硬规则 Gate（不可被 LLM 绕过）

## 2. 状态机

```
                    ┌──────────────────────────────────┐
                    │              INIT                  │
                    │  解析 Task，提取 symbol / period   │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │         LOAD_CONTEXT              │
                    │  加载 Memory / 用户偏好 / 上次复盘 │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │         SELECT_SKILL              │
                    │  匹配 Skill，加载 Skill 定义        │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │            PLAN                   │
                    │  生成执行计划（工具+Agent 调用顺序）│
                    │  计划必须符合 Skill.hard_rules     │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │         EXECUTE_STEP              │
                    │  执行单步（Tool 调用 或 Agent 分析）│
                    │  每步写入 Trace                    │
                    │  检查剩余步数/预算/超时            │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │        VALIDATE_RESULT            │
                    │  Schema 校验 + 数据完整性检查       │
                    │  失败 → 重试（最多 1 次）          │
                    │  失败 → 标记为 INCOMPLETE          │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │        COLLECT_EVIDENCE            │
                    │  汇总所有 evidence                 │
                    │  去重，生成 evidence_id             │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │           ANALYZE                  │
                    │  调用 Agent 角色分析证据           │
                    │  多头 + 空头 + 财务 + 风控          │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │            JUDGE                  │
                    │  风控裁决官给出最终决策             │
                    │  硬规则 Gate: 强制检查             │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │           REPORT                  │
                    │  生成报告（Markdown/结构化）        │
                    └──────────────┬───────────────────┘
                                   │
                    ┌──────────────▼───────────────────┐
                    │        WRITE_MEMORY               │
                    │  写入决策日志、复盘记录             │

                                   │
                    ┌──────────────▼───────────────────┐

                    └──────────────────────────────────┘
```

## 3. EXECUTE_STEP 子状态

EXECUTE_STEP 是核心执行状态，内部有微循环：

```
EXECUTE_STEP
  ├─ 判断当前步骤类型
  │   ├─ TOOL_CALL → Tool Router → 返回 ToolResult
  │   ├─ BASH_CMD → Bash Executor → 返回 BashResult
  │   └─ AGENT_CALL → Model Gateway → 返回 AgentOutput
  ├─ 写入 Trace
  ├─ 检查:
  │   ├─ 硬规则触发? → 立刻 JUDGE (跳过后续)
  │   ├─ 步数超限? → 立刻 JUDGE (标记 TRUNCATED)
  │   ├─ 预算超限? → 立刻 JUDGE (标记 BUDGET_EXCEEDED)
  │   └─ 超时? → 立刻 JUDGE (标记 TIMEOUT)
  └─ 继续下一步 (循环到 PLAN 中的所有步骤)
```

## 4. 硬规则 Gate

在 JUDGE 阶段，硬规则优先级高于 LLM 判断：

1. **ST / 退市** → EXCLUDE（不可覆盖）
2. **数据不足** → CAUTIOUS_HOLD（不可推为 HOLD）
3. **现金流连续 2 季恶化** → CAUTIOUS_HOLD 或 REDUCE_EXIT
4. **投资假设被证伪** → REDUCE_EXIT（不可推为 HOLD）

如果 LLM 给出的 action 与硬规则冲突，系统**以硬规则为准**，并在 output 中标记 `hard_rule_override: true`。

## 5. Loop 终止条件

以下任一条件满足时，Loop 必须终止：

- PLAN 中的所有步骤完成
- 硬规则触发 CRITICAL 风险
- 步数超限
- LLM 调用次数超限
- 超时
- 预算耗尽
- FINISH 状态达成

## 6. 错误处理

- 单步工具调用失败 → 重试一次（不同参数），仍失败 → 标记为 UNAVAILABLE
- LLM 调用失败 → Fallback 到 Flash 模型，仍失败 → 标记为 MODEL_UNAVAILABLE
- 状态机死锁检测 → 强制 FINISH

## 7. 不允许的行为

- ❌ 无限循环（无上限的 ReAct）
- ❌ LLM 自行发明工具调用
- ❌ 跳过硬规则 Gate
- ❌ 绕过步数/超时限制
- ❌ 在 EXECUTE_STEP 内做"再想一次"
- ❌ 无 trace 记录的工具调用
